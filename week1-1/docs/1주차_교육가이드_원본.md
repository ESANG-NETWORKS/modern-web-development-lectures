# 1주차 교육 가이드 (2회차 × 1시간)

## 문서 목적

- 1주차에 진행할 Git/GitHub 협업 교육을 **바로 실행 가능한 형태**로 정리한다.
- 교육 핵심은 다음 3가지다.
  1. VS Code에서 Git 기초 기능을 익힌다.
  2. GitHub Flow 기반으로 개발/리뷰/병합 과정을 이해한다.
  3. `docs/uml` 브랜치 산출물을 팀에 공유하고 피드백을 받는다.
- 분량이 허용되면 VS Code에서 GitHub Copilot 기본 사용법을 짧게 소개한다.

---

## 운영 개요

- 대상: 프로젝트 참여 개발팀 3명
- 횟수/시간: 주 2회, 회차당 60분
- 진행 방식: 20% 이론 + 80% 실습
- 준비물
  - VS Code 설치
  - Git 설치 및 계정 인증 완료
  - 원격 저장소 접근 권한 확인
  - 필수 확장: GitHub Pull Requests and Issues, GitLens(선택), GitHub Copilot(선택)

## Session 1 (60분) — VS Code로 배우는 Git 기초

### 학습 목표 (Session 2)

- VS Code Source Control 화면에서 변경 사항을 확인하고 커밋할 수 있다.
- 브랜치를 생성/전환하고, 원격에 Push/Pull 할 수 있다.
- 충돌이 발생했을 때 VS Code의 Merge Editor로 기본 해결 흐름을 경험한다.

### 시간표 (Session 2)

#### 0~10분: 왜 Git을 쓰는가 + 오늘 실습 흐름 안내

- 로컬 이력 관리, 협업 충돌 최소화, 롤백 가능성 강조
- 오늘 완성 기준 안내
  - `feature/` 브랜치 생성
  - 문서 1개 수정 후 커밋/푸시
  - 간단 충돌 상황 재현 후 해결

#### 10~25분: VS Code Source Control 기본 조작

- 실습 항목
  - 변경 파일 확인 (`M`, `U` 상태 읽기)
  - Stage / Unstage
  - 커밋 메시지 작성
  - 커밋 히스토리 확인 (Timeline 또는 Git Graph)
- 커밋 메시지 예시
  - `docs: 1주차 교육 가이드 초안 추가`
  - `fix: uml 용어 오타 수정`

#### 25~45분: 브랜치/원격 기본 실습

- 실습 순서
  1. `develop` 최신화 (`Pull`)
  2. `feature/git-basic-{이름}` 브랜치 생성
  3. 파일 수정 후 커밋
  4. 원격으로 `Push` 및 Upstream 연결
  5. 브랜치 전환 후 변경 상태 확인
- 체크포인트
  - 각자 브랜치가 원격에 생성되었는지 확인
  - 커밋 1건 이상 생성 확인

#### 45~55분: 충돌(Conflict) 미니 실습

- 같은 파일 같은 줄을 2개 브랜치에서 수정해 충돌 유도
- VS Code Merge Editor에서 해결
  - Current / Incoming / Both 선택
  - 수동 편집 후 충돌 마커 제거
  - 해결 커밋 생성

#### 55~60분: 정리 및 과제

- 오늘 배운 명령/동작 3가지 구두 확인
- 과제
  - 각자 실습 브랜치에서 문서 1회 추가 수정
  - 다음 회차 전 원격 Push 완료

## Session 2 (60분) — GitHub Flow 실전 + docs/uml 공유

### 학습 목표

- GitHub Flow 기준으로 이슈 단위 브랜치 → PR → 리뷰 → 머지 흐름을 수행할 수 있다.
- 팀 브랜치인 `docs/uml`에서 만든 내용을 공유하고 개선 포인트를 도출한다.
- 남는 시간에 Copilot으로 반복 작업을 줄이는 방법을 체험한다.

### 시간표

#### 0~10분: GitHub Flow 핵심 규칙 정리

- 단순 규칙 4가지
  1. `main`(또는 보호 브랜치)은 항상 배포 가능 상태 유지
  2. 작업은 짧은 수명의 기능 브랜치에서 진행
  3. PR로 리뷰/논의 후 병합
  4. 병합 후 브랜치 정리
- 팀 적용안
  - 브랜치: `feature/*`, `docs/*`, `hotfix/*`
  - 리뷰 최소 1인 승인

#### 10~30분: PR 실습 (VS Code + GitHub)

- 실습 순서
  1. `feature/flow-pr-{이름}` 브랜치 생성
  2. 작은 변경 1건 커밋/푸시
  3. PR 생성 (제목/본문 템플릿 사용)
  4. 리뷰 코멘트 남기기 및 수정 반영
  5. Squash Merge 또는 Rebase Merge로 병합
- PR 본문 예시 템플릿
  - 변경 내용
  - 확인 방법
  - 리뷰 포인트

#### 30~45분: `docs/uml` 브랜치 공유 세션

- 공유 목표
  - 브랜치 산출물의 구조/용어/흐름을 팀이 동일하게 이해한다.
- 진행 방식
  1. 작성자 5분 발표: 무엇을 왜 이렇게 모델링했는지
  2. 팀 7분 피드백: 누락/중복/모호 용어 점검
  3. PM 3분 정리: 최종 반영 우선순위 지정
- 공유 체크리스트
  - 용어가 기능 명세와 일치하는가?
  - 화면/도메인/데이터 흐름 연결이 자연스러운가?
  - 다음 구현 스프린트에서 바로 사용할 수 있는가?

#### 45~55분: (옵션) VS Code GitHub Copilot 10분 입문

- 최소 교육 범위
  - 인라인 제안 수락/부분 수락
  - Chat에 코드 맥락 질문하기
  - 문서 초안/커밋 메시지 초안 생성
- 실습 프롬프트 예시
  - `이 변경 내용을 Conventional Commit 형식으로 요약해줘.`
  - `이 PR 설명을 리뷰어가 이해하기 쉽게 3줄로 정리해줘.`
- 주의사항
  - 생성 코드는 반드시 리뷰 후 반영
  - 민감 정보(실제 비밀번호/키) 프롬프트 입력 금지

#### 55~60분: 회고 & 다음 주 연결

- 각자 한 문장 회고
  - Keep / Problem / Try 1개씩
- 다음 주 예고
  - API 개발 시 PR 단위 쪼개기
  - 리뷰 코멘트 응답 규칙 적용
